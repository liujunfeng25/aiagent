## 二、技术领域

涉及计算机软件、数据处理、Web 实时通信与机器学习应用技术领域，尤其适用于 **餐饮 / 农产品 / 商贸供应链** 的 **数字化运营与智能稽核** 场景。

## 三、背景技术（交底用）

- 企业已有 ERP 或交易系统，订单与商品主数据在 **业务数据库** 中维护；重复建设一套写入型「第二 ERP」成本高且风险大。  
- 管理层需要 **大屏、驾驶舱、实时经营曲线**，但直接在生产库上高频写入或改表结构（DDL）难以通过运维与合规审批。  
- AI 能力（图像识别、单据识别）常分散在独立工具，缺乏与 **统一门户、模型版本、部署状态** 联动的平台。

---

## 四、要解决的技术问题（归纳）

1. 如何在 **不对业务库做写入、尽量不改表结构** 的前提下，提供 **可靠的 KPI 聚合与近实时推送**？  
2. 如何在 **多租户、表名/主键不统一** 的 MySQL 环境中 **低代码对接** 订单与明细？  
3. 如何形成 **数据集—训练—模型部署—在线推理** 的可运维闭环？  
4. 如何将 **票据结构化结果** 与 **订单数据** 进行可配置的 **比对与展示**？

---

## 五、技术方案概述（与代码对应）

### 5.1 只读洞察层

- 配置业务库只读账号；分析 API 限制 **SELECT** 语句。  
- 订单相关查询通过 **schema 白名单 + 元数据探测** 适配 `orders` / `orders_items` 等差异。

### 5.2 增量轮询与 WebSocket 广播

- 在内存中维护 **当日累计指标** 与 **水印**（如时间与主键），按间隔 **增量查询** 业务订单表。  
- WebSocket 会话接入后下发 **snapshot**；周期性 **ping**；指标更新时向连接集合 **广播** JSON 消息（如 `ping` / `snapshot` / `batch`）。  
- **效果**：前端驾驶舱/天枢获得「实时链路」观感，而业务库侧仍为 **只读查询**。

### 5.3 AI 训练与推理平台化

- SQLite 记录数据集、任务、模型；子进程训练写状态文件；模型 **部署标记** 驱动识别路由默认引擎。  
- 支持 **预置模型** 一键注册，缩短上线周期。

### 5.4 票据与比对

- 图像上传后经识别引擎输出 **结构化字段**；可选 **SSE** 推送长耗时进度。  
- 与订单结构化数据 **比对**，输出差异（具体规则由实现配置）。

### 5.5 智能物流（可选写入点）

- 物流模块使用 **独立租户库** 连接策略，与既有 PHP 业务 **库名切换** 行为对齐；地图展示涉及 **坐标系转换**。

## 六、有益效果（答辩/申报用）

- **风险可控**：核心经营数据以 **只读** 方式消费，降低对生产交易系统的侵入。  
- **部署灵活**：**零 DDL** 实时通道适配运维「不改表」约束。  
- **对接成本低**：表名/主键自动探测减少定制开发。  
- **AI 可运营**：训练与推理 **版本化、可部署、可切换**。  
- **合规叙事清晰**：适合 **央企二级公司** 「数字化转型 + AI 赋能 + 自主研发」材料。
